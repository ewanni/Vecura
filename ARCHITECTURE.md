# Архитектурный документ: Vecura (Wails v2 + RAM Vector Search)

Переход к архитектуре с In-Memory поиском кардинально упрощает кроссплатформенную сборку. Мы изолируем «токсичный» CGO исключительно в модуле локальной генерации эмбеддингов (`llama.cpp`), а база данных и алгоритмы поиска становятся на 100% нативными для Go.

Ниже представлен детальный проект монолитного приложения, оптимизированного под максимальную производительность сборщика мусора (GC) в Go. Документ включает пересмотренные разделы с учётом динамических моделей (разный `Dim`), добавления моделей по API и использования удалённых провайдеров (OpenRouter / OpenAI-совместимых).

---

## 1. Обновленный технологический стек

| Слой | Выбранная технология | Обоснование |
| --- | --- | --- |
| **GUI Framework** | `wailsapp/wails/v2` | Связывает Go-бэкенд и фронтенд (Vue/Svelte) через легковесный IPC. |
| **Embedder (Local LLM)** | `go-skynet/go-llama.cpp` | Единственный модуль, требующий CGO. Загружает GGUF в память, генерирует векторы переменной размерности. |
| **Embedder (Remote API)** | `net/http` (OpenAI-совместимый клиент) | **Pure Go (без CGO).** OpenRouter / OpenAI / совместимые провайдеры. |
| **Database** | `modernc.org/sqlite` | **Pure Go (без CGO).** Хранит пути, теги, метаданные и сырые BLOB-векторы. |
| **Vector Engine** | Кастомный In-Memory (Model-Aware) | Многопоточный расчет косинусного сходства в оперативной памяти. |

## 2. Схема базы данных (SQLite, pure-Go)

Векторы хранятся отдельно от метаданных, так как одна картинка может иметь эмбеддинги от **нескольких моделей/провайдеров**, и размерность `Dim` у них разная. Векторы хранятся в базе исключительно для персистентности (чтобы не пересчитывать их при перезапуске программы).

```sql
CREATE TABLE images (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    file_path  TEXT UNIQUE,
    prompt     TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- Таблицы для жестких тегов остаются для гибридного поиска
CREATE TABLE tags (
    id   INTEGER PRIMARY KEY,
    name TEXT UNIQUE
);

CREATE TABLE image_tags (
    image_id INTEGER,
    tag_id   INTEGER,
    PRIMARY KEY (image_id, tag_id)
);
CREATE INDEX idx_image_tags_tag ON image_tags(tag_id);

-- Один эмбеддинг = одна (картинка, провайдер + модель)
-- Модель идентифицируется парой (provider, model_id):
--   локальный llama.cpp и удалённый OpenAI дают РАЗНЫЕ векторные пространства,
--   их нельзя смешивать в одном поиске.
CREATE TABLE embeddings (
    image_id   INTEGER,
    provider   TEXT,           -- 'local-llama' | 'openai' | 'openrouter' | ...
    model_id   TEXT,           -- 'nvidia/nemotron' | 'text-embedding-3-small' | ...
    dim        INTEGER,        -- 768 | 1024 | 2048 | 4096  (для валидации)
    vector     BLOB,           -- []float32, длина = dim*4 байт
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (image_id, provider, model_id)
);
CREATE INDEX idx_embeddings_model ON embeddings(provider, model_id);
```

**Ключевое изменение:** модель теперь идентифицируется парой `(provider, model_id)`. Поиск всегда привязан к конкретной паре, и загрузка boot-кода читает `dim` из БД, а не хардкодит 2048.

## 3. Архитектура памяти (Model-Aware Flat Store)

`Dim` больше не константа. Векторный движок — это **набор flat-array сторов, по одному на каждую пару (provider, model_id)**.

```go
type ModelVectors struct {
    Dim     int       // 768 / 1024 / 2048 / 4096
    Cap     int       // выделенная ёмкость (элементов, не байт)
    Count   int       // фактически заполнено
    IDs     []int32   // image_id, длина Count
    Vectors []float32 // плоский, длина Cap*Dim; используется [:Count*Dim]
    InvNorm []float32 // 1/|V| для каждого вектора (см. критику №2)
}

type VectorStore struct {
    mu     sync.RWMutex
    models map[string]*ModelVectors          // key = provider + "/" + model_id
    snap   atomic.Pointer[storeSnapshot]     // read-only snapshot для поиска
}

type storeSnapshot struct {
    models map[string]*ModelVectors
}
```

### Изменения против оригинала

1. **Preallocate + `Count`, без `append`.** Добавление растит `Cap` батчами (×1.5) под `Lock()`, но чтение поиска идёт через **atomic snapshot** (`atomic.Pointer`), а не `RLock` на весь поиск. Читающие горутины получают неизменяемый снимок → нет блокировки на 20–30 мс, нет race при `memmove` (см. критику №1). После добавления делается `snap.Store(newSnapshot)`.

2. **Динамический `Dim`.** Boot-загрузка читает `dim` из БД, группирует векторы по `(provider, model_id)` и строит отдельный `ModelVectors` на каждую группу. Нормализация (`InvNorm`) считается **один раз при записи** в БД и кэшируется — знаменатель в формуле косинуса не пересчитывается на каждый поиск (см. критику №2).

3. **Декод BLOB без лишних аллокаций.** BLOB из `modernc.org/sqlite` преобразуется в `[]float32` через `unsafe.Slice((*float32)(unsafe.Pointer(&blob[0])), len(blob)/4)` — без копии, без GC-pressure при boot (см. критику №4).

* **Инициализация (Boot):** При старте приложения Go выполняет `SELECT image_id, provider, model_id, dim, vector FROM embeddings`. BLOB конвертируется в `[]float32` (без копии) и дописывается в соответствующий плоский массив `Vectors` выбранного `ModelVectors`.
* **Потребление RAM:** Для 100,000 картинок и `Dim=2048` массив `Vectors` займет ровно один непрерывный блок памяти в **~800 МБ** (плюс `IDs` ~0.4 МБ, `InvNorm` ~0.4 МБ). Для GC это выглядит как *один* объект, не содержащий указателей. Время сканирования GC равно нулю. При другом `Dim` объём = `N * Dim * 4` байт (см. раздел 8).

### Boot (параллельный decode)
```go
rows := db.Query("SELECT image_id, provider, model_id, dim, vector FROM embeddings")
// группируем по ключу, параллельно декодим BLOB -> float32, пишем в ModelVectors
```

## 4. Пайплайн семантического поиска

Поиск ближайших соседей основан на вычислении косинусного сходства между вектором запроса $Q$ и вектором из базы $V$:

$$\text{similarity} = \frac{\sum_{i=1}^{n} Q_i V_i}{\sqrt{\sum_{i=1}^{n} Q_i^2} \sqrt{\sum_{i=1}^{n} V_i^2}} = \text{Dot}(Q, V) \cdot \text{InvNorm}[V]$$

### Алгоритм исполнения запроса

1. **Пользователь ищет:** "Cyberpunk city".
2. **Инференс:** Строка отправляется в активный `Embedder` (локальный через CGO в `llama.cpp`, либо удалённый через HTTP к OpenRouter/OpenAI). Модель возвращает вектор запроса `Q` (размерности `Dim` выбранной пары `provider/model_id`, L2-нормализованный).
3. **Выбор стора:** Поиск выбирает `ModelVectors` по ключу `(provider, model_id)` из snapshot. Смешивание разных пространств невозможно — несовместимые векторы физически разделены.
4. **Параллельный расчет:**
   * Длина плоского массива `Vectors` делится на количество ядер CPU (`runtime.NumCPU()`).
   * Запускаются горутины, каждая из которых вычисляет скалярное произведение для своего сегмента, умножая на прекомпьютед `InvNorm` (знаменатель уже = 1 для запроса, норма базы учтена заранее).
5. **Top-K Сортировка (локальные кучи + merge):** Каждая горутина ведёт **свой локальный Min-Heap** фиксированного размера K (например, 50). После завершения сегментов кучи сливаются (merge) в финальный Top-K. Это устраняет узкое место общего mutex (см. критику №3).
6. **Ответ:** Полученные ID изображений передаются в `modernc.org/sqlite` для извлечения путей и формирования ответа для Wails. Весь поиск по 100k картинок занимает менее 20–30 мс (при локальном инференсе; remote-вызов добавляет сетевую латентность).

## 5. Слой провайдеров эмбеддингов

Документ изначально утверждал, что CGO нужен только для `llama.cpp`. С появлением API-провайдеров картина уточняется:

```go
type Embedder interface {
    Key() string                    // "local-llama/nemotron" | "openai/3-small"
    Dim() int
    Embed(texts []string) ([][]float32, error)
}

// Локальный — единственный CGO-путь
type LocalLlamaEmbedder struct{ model *llama.Model } // go-skynet / llama.cpp

// Удалённые — чистый Go (net/http), без CGO
type OpenAICompatibleEmbedder struct {
    BaseURL string // https://openrouter.ai/api/v1  |  https://api.openai.com/v1
    APIKey  string
    Model   string
}
```

- **CGO остаётся только у `LocalLlamaEmbedder`** — изоляция сохраняется. OpenRouter/OpenAI — pure Go HTTP, не требуют CGO и не создают C-утечек.
- **Добавление модели по API** = регистрация нового `OpenAICompatibleEmbedder` в registry (BaseURL + APIKey + Model). Для локальных моделей — `llama_load_model` через **менеджер моделей** (load/unload, RAM-budget, LRU-eviction, валидация GGUF-файла).
- **Ключевой риск remote-провайдеров:** rate limits / latency / стоимость при сканировании папки из 100k файлов. Нужен батчинг (batch embed), очередь и кэш по `hash(file + prompt + model)`, иначе сканирование встанет или сожжёт бюджет.

### Менеджер моделей (lifetime + RAM-budget)
- Registry `map[modelKey]*LoadedModel` под `sync.RWMutex` (чтение при инференсе, запись при add/unload).
- Лимит одновременно загруженных **локальных** моделей; LRU-eviction при нехватке RAM (сами GGUF весят 2–8 ГБ каждая).
- Выгрузка обязательна: `llama_free_model` для локальных; remote-клиенты stateless.

## 6. Подводные камни и стратегии смягчения

### А. Синхронизация при добавлении новых файлов
**Уязвимость:** Watcher сканирует новые картинки, и новые векторы нужно добавить в In-Memory хранилище в момент активного поиска.
**Решение:** Atomic snapshot вместо `RLock` на весь поиск.
* Поиск читает `snap.Load()` — неизменяемый снимок, множественные одновременные поиски без блокировки.
* Добавление под `Lock()` растит `Cap`/пишет в `Vectors[:Count*Dim]`, затем `snap.Store(newSnapshot)`. Добавление — микросекунды, UI не зависнет.

### Б. Ограничения Wails IPC (Изображения)
**Уязвимость:** Передача сырых `[]byte` картинок из Go во фронтенд через события Wails вызовет катастрофические лаги сериализации JSON и падение (OOM).
**Решение:**
1. Отдавать превью через метод Go-бэкенда (data-URI base64 для 256×256 webp — ~10–30 КБ, сериализация дешёвая) либо монтировать папку thumbnails как статику. Кастомный `wails://images/` протокол — переусложнение (см. критику №6).
2. В пайплайне воркера-сканера генерировать легковесные превью `.webp` (256×256) в скрытую папку (например, `~/.vecura/thumbnails/`).
3. Фронтенд запрашивает только пути/URI к thumbnail-версиям.

### В. CGO Утечки памяти (Memory Leaks)
**Уязвимость:** Эмбеддинги локально генерируются через C-код `llama.cpp`. Забытый указатель положит систему при сканировании крупной папки.
**Решение:** Строгая изоляция инференса.
```go
cVector := C.llama_generate_embedding(...)
defer C.free(unsafe.Pointer(cVector)) // Обязательно освобождаем C-память
// Копирование C-массива в Go-срез ДО выхода из функции
```
Remote-провайдеры (OpenAI/OpenRouter) не имеют C-памяти — утечки возможны только в виде незакрытых HTTP-соединений (используйте `http.Client` с `Transport.MaxIdleConns`).

### Г. Персистентность при добавлении
**Уязвимость:** Добавление в RAM под `Lock()` не пишет в SQLite — падение процесса между добавлением и коммитом теряет данные.
**Решение:** Явный `INSERT` в таблицу `embeddings` внутри WAL-транзакции с батчингом (bulk insert). Boot перечитывает `embeddings` целиком.

### Д. Гибридный поиск (semantic + tag filter)
**Уязвимость:** Фильтрация по тегу без индекса при 100k строках = полный скан.
**Решение:** Индекс `idx_image_tags_tag` уже добавлен. Поиск сначала получает множество `image_id` по тегу через `JOIN`, затем ограничивает косинусный перебор только этим множеством (или наоборот: Top-K, затем фильтр по тегу — зависит от селективности).

## 7. Метрики памяти (реалистичные)

Заявленные 800 МБ — только векторы при `Dim=2048`. Реальный footprint:

| Компонент | Объём |
| --- | --- |
| Векторы (100k, Dim=768) | ~300 МБ |
| Векторы (100k, Dim=2048) | ~800 МБ |
| Векторы (100k, Dim=4096) | ~1.6 ГБ |
| Локальный GGUF (на модель) | 2–8 ГБ |
| Превью webp (100k × ~20 КБ) | ~2 ГБ на диске |
| BLOB в SQLite (зеркало векторов) | равно объёму векторов |

При нескольких одновременно загруженных локальных моделях и нескольких `Dim` реальный RAM-расход легко **10–20 ГБ**. Документ вводит явный **бюджет RAM** и политику eviction моделей (см. раздел 5).

## 8. Критические замечания к исходному проекту (сводка)

1. **`append` в плоский массив ломает конкурентность** — `append` может сменить базовый указатель и вызвать `memmove` во время чтения. Заменено на preallocate + `Count` + atomic snapshot.
2. **«Знаменатель = 1» неверно для базы** — нормализация при записи + прекомпьютед `InvNorm[]`.
3. **Общий Min-Heap под mutex убивает параллелизм** — локальные кучи на сегмент + merge.
4. **BLOB→`[]float32` копия создаёт GC-pressure при boot** — `unsafe.Slice` поверх BLOB без копии.
5. **Watcher + RWMutex не решает персистентность** — WAL-транзакционный batch-insert.
6. **Кастомный `wails://` протокол — переусложнение** — data-URI / статика для превью.
7. **Нет индекса по тегам** — добавлен `idx_image_tags_tag`.
8. **800 МБ — только векторы** — реальный footprint 3–4× (модели, превью, BLOB).

### Дополнительно (динамические модели + API)
9. **`Dim=2048` жёстко зашит** — теперь `Dim` динамический, сторы разбиты по `(provider, model_id)`.
10. **Модели добавляются по API** — нужен менеджер моделей (lifetime, RAM-budget, LRU-eviction, валидация GGUF) и явное разделение remote (pure Go HTTP) vs local (CGO).
11. **Remote-провайдеры (OpenRouter/OpenAI)** — чистый Go, но требуют батчинга, очереди и кэша из-за rate limits / стоимости при массовом сканировании.
12. **Один BLOB на image несовместим с несколькими моделями** — вынесено в таблицу `embeddings(image_id, provider, model_id, dim, vector)`.
